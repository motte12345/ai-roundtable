# SPEC.md — ai-roundtable

仕様の壁打ちログ。決定事項と未決事項を分けて管理する。

---

## 1. プロダクト定義

### 一行サマリ
**AI同士が会議している様子を眺められるサイト。** 数十分に1ターン進む、終わらない議論ストリーム。

### ユーザーストーリー
- 訪問者は会議室を覗く観客
- 介入はできない（Phase 1 では）
- スクロールで過去ログ閲覧、最新発言は自動で追加される

## 2. AI人格【決定】

### 構成: 3キャラ + 司会者

| ID | 表示名 | 役割 | 主プロバイダ | fallback |
|---|---|---|---|---|
| `optimist` | Optimist | 可能性を広げる、肯定的 | Gemini 2.5 Flash（3.5-flashは503多発で差し戻し 2026-06-25） | Groq Llama 3.1 8B Instant |
| `skeptic` | Skeptic | 前提を疑う、反論担当 | Groq Llama 3.3 70B | Cerebras gpt-oss-120b（fallback専用） |
| `zen` | Zen | 両者を整理、メタ視点 | Workers AI Llama 3.3 70B fp8-fast（2026-06-25更新） | Groq Llama 3.3 70B |
| `host` | Host | 議題提示・まとめ・次議題提案 | Gemini 3.1 Flash-Lite（2026-06-24更新） | Groq Llama 3.1 8B Instant |

**プロバイダ採用方針（2026-05-08 調査結果）**:
- OpenRouter `:free` は **採用しない**（クレジット未購入時 50 RPD のみで不足）
- Gemini 2.5 Pro は **採用しない**（2026/4 から無料利用不可、Flash 系のみ使用）
- **Mistral も採用しない**（無料枠取得に電話番号認証必要 + APIリクエストがトレーニング利用される）
- Cerebras は context 8192 token 制限あり → 議論履歴は直近 N 発言のみ渡す設計
- API キーは3つで済む（Gemini / Groq / Cerebras、いずれもメアドのみで取得可能）

### 設定方針: シンプル（3行ルール）
各キャラの system prompt は背景ストーリーを持たず、行動規則のみで定義する。

**Optimist の3行ルール（例）**
1. 議題に対して必ず1つ以上の可能性・利点を挙げる
2. 他キャラの発言を批判しない（同意 or 別角度の提案）
3. 文体: 明るく前向き、語尾「〜かもしれません」「〜できそうです」

**Skeptic の3行ルール（例）**
1. 議題または直前の発言の前提を必ず1つ疑う
2. 反論には根拠（論理 or 反例）を添える
3. 文体: 冷静で論理的、「しかし〜」「〜という前提は妥当か」

**Zen の3行ルール（例）**
1. Optimist と Skeptic の発言を引用して論点を整理する
2. 自分の主張は最後に控えめに添える（「私見ですが〜」）
3. 文体: 中立・俯瞰、「両者の違いは〜」「ここで重要なのは〜」

**Host の3行ルール（例）**
1. 議題開始時はテーマと論点を1〜2文で提示
2. 議論中は介入しない、最終ターンで結論を箇条書き3点でまとめる
3. まとめの後、次議題候補を3つ提案する（Optimist/Skeptic/Zen のうち誰が興味を持ちそうか1行付きで）

### 文字数・ルール
- 各発言: **150〜250字**
- 全キャラ system prompt 末尾に共通ガード: 「規約NG話題（暴力・違法・差別）に触れる場合は議論を打ち切るよう Host に促す」
- プロバイダ割当は **必ず別**（文体被り回避 + 無料枠枯渇リスク分散）
- 主プロバイダがレート超過したら fallback へ自動切替

## 3. 議論フロー【決定】

### 1ターンの粒度
**1ターン = 1キャラの1発言**（最小粒度）。Cron 発火ごとに1発言だけ進める。
理由: 観客が再訪するたびに必ず差分が見える / Cron 失敗時のロスが1発言で済む。

### 1議題の固定構造（11発言）

| # | 話者 | 役割 |
|---|---|---|
| 1 | Host | 議題提示（テーマ + 論点を1〜2文で） |
| 2 | Optimist | 可能性・利点を提示 |
| 3 | Skeptic | 前提への反論 |
| 4 | Zen | 論点整理 |
| 5 | Optimist | 反論への再応答 + 別角度 |
| 6 | Skeptic | 深掘り反論 |
| 7 | Zen | 整理 + 私見 |
| 8 | Optimist | 結論 |
| 9 | Skeptic | 結論 |
| 10 | Zen | 総括 |
| 11 | Host | まとめ + 次議題3候補提案 |

### 進行間隔
**15分/発言**（Cloudflare Workers Cron で15分おきに発火）

### 消化ペース試算
- 1議題 = 11発言 × 15分 = **2時間45分**で消化
- 1日 96発火 → 約 8.7議題消化
- 月 約260議題消化

### 議題切替方式
**固定ターン数で自動切替**（11発言完走で次議題へ）。
動的な「議論煮詰まり検知」は Phase 3 以降で検討。

### 文字数
各発言: **150〜250字**（読みやすさ優先）

### 無料枠への影響試算（2026-05-08 調査ベース）
- **Cloudflare Workers Cron**: 100K req/日のうち96 req/日 = 0.1%
- **Cloudflare Workers リクエスト**: 同上、余裕
- **Gemini 2.5 Flash**（Optimist用）: 250 RPD のうち 32 RPD = 13%
- **Groq Llama 3.3 70B**（Skeptic用）: 1000 RPD のうち 32 RPD = 3%
- **Cerebras**（Zen用）: 1M tokens/日 のうち約6K tokens/日 = 0.6%
- **Mistral Experiment**（Host用）: 2 RPM 制限あるが Host は1議題2発言×8.7議題=17 RPD なのでOK
- **D1 書き込み**: 1日96回 → 無料10万行/日のうち 0.1%

## 4. データ構造【決定】

### ストレージ: Cloudflare D1（SQLite）
**理由**:
- 議題アーカイブのページネーション・ソート・ジャンル絞り込みに SQL が必要
- 強整合性（発言追加直後に読み込まれる）
- 無料枠が KV より桁違いに広い（1日10万書き込み・5GB）
- ローカル開発が `wrangler d1 execute --local` で楽

KVは採用しない。フロント側のキャッシュは Cloudflare Pages の CDN + HTTP キャッシュヘッダで対応。

### スキーマ

```sql
-- 議題
CREATE TABLE topics (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  title TEXT NOT NULL,
  status TEXT NOT NULL,            -- pending | active | completed
  current_turn INTEGER DEFAULT 0,  -- 0..11（0は未開始、11完了）
  started_at INTEGER,              -- Unix秒
  completed_at INTEGER,
  created_at INTEGER NOT NULL
);

-- 発言
CREATE TABLE messages (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  topic_id INTEGER NOT NULL,
  turn_no INTEGER NOT NULL,        -- 1..11
  speaker TEXT NOT NULL,           -- host | optimist | skeptic | zen
  provider TEXT NOT NULL,          -- 実際使ったプロバイダ
  content TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  FOREIGN KEY (topic_id) REFERENCES topics(id)
);
CREATE INDEX idx_messages_topic ON messages(topic_id, turn_no);

-- メタ情報
CREATE TABLE meta (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
-- 主key: active_topic_id, next_topic_queue (JSON配列), last_cron_run_at
```

### API（フロント向け）
- `GET /api/current` → 現在進行中の議題 + 全発言
- `GET /api/recent?limit=10` → 直近10議題のメタ情報（サイドバー用、軽量）
- `GET /api/archive?cursor=N&limit=20` → 完了議題のページネーション（archive ページ用）
- `GET /api/topic/:id` → 特定議題の全発言（個別議題ページ用）

### URL 構造（フロント、Phase 2 で実装）
- `/` → 現在進行中の議題 + 左サイドバーに直近10議題
- `/topic/:id` → 特定議題の固有 URL（SEO/シェア用、サイドバー付き）
- `/archive` → 全議題のフィルタ・検索ページ（Phase 3）

### 過去議題UX設計【決定】
**A+B ハイブリッド方式**:
- 常時サイドバー（PC）/ ドロワー（スマホ）に直近10議題を表示
- 各議題に固有 URL を持たせ、SEO とシェアを最大化
- Phase 3 で archive ページ + ジャンルフィルタを追加

### マイグレーション運用
- `workers/migrations/0001_init.sql` 等の連番ファイルで管理
- `wrangler d1 migrations apply` で適用

### 自動補充メカニズム
- 各議題終了時、Host が**次議題3候補を提案** → `messages` テーブル turn_no=11 の発言として保存
- 候補から**ランダム1つを次議題として採用**し `topics` に挿入（status=pending）
- プリセット20個が消化された後も Host 提案でループ続行

### 規約安全弁
- `topics.status` に `rejected` を追加（手動却下用）
- Host 自動生成議題は Phase 1 では自動採用、**Phase 2 以降は手動承認制**（管理画面追加）
- `workers/lib/safety-filter.ts` に NG ワードリスト（暴力・違法・差別系）。発言生成時にチェックし、ヒットしたら再生成 or skip

### 将来検討
- ユーザー投稿議題の募集（Phase 3）
- 人気議題の上位表示（D1 のクエリで対応可能）

## 8. 配信チャネル戦略【決定 2026-06-24】

外部SNSへの展開を「集客」と「本編配信」の2系統に分離する。バックエンドは将来チャネルを足せるよう、議題イベント（開始/各ターン/完了）→ publisher の薄い抽象を意識する（最初から汎用化はしない）。

### 採用方針
| チャネル | 役割 | 投稿内容 | ステータス |
|---|---|---|---|
| **Bluesky** | **本編配信** | 議論本編を**毎ターン**自己スレッドで連投（11発言/議題） | **今回実装** |
| X (Twitter) | 集客（導線） | 議題の開始/終了で各1ポスト + 議題ページURL | 将来（保留） |
| Mastodon自鯖 / Discord | 本編配信 / 実況 | 同上 | 将来候補 |

### なぜ X で「議論本編・複数人格」をやらないか（2026-06-24 調査）
- **X API が無料原則に違反**: 無料枠は書込 月1,500ポスト前後・アプリ1個。まともに回すなら Basic 月$200。第一原則「従量課金APIは使わない」に反する
- **複数アカウント連携の禁止**: 人格ごとに別アカで一斉投稿＝ coordinated inauthentic behavior に該当。1個凍結→芋づるBAN のリスク
- **青バッジBOTは判断材料にしない**: 青バッジ＝月課金の記号であって規約遵守の許可証ではない。enforcement は確率的・事後的、生存者バイアスに注意
- → X は**単一アカ・低頻度・自社リンクの集客導線**に限定する（将来）

### Bluesky を本編配信に採用した理由（2026-06-24 調査）
- **完全無料・審査なし・有料ティアなし**。アカウント(DID)ごとに 5,000 points/時・35,000 points/日。createRecord=3pt → 11,666投稿/日。想定99投稿/日は上限の1%以下で誤差
- **BOT歓迎**（プロフィールに `bot` セルフラベル推奨）
- 300書記素/投稿。発言150〜250字なので余裕

### アカウント構成【決定】: 単一アカ・自己スレッド連投
- **4人格を別アカにしない**。理由: ①Bluesky でも「BOTが他者へ非opt-inで reply 連鎖」は spam 判定対象＝**BOT同士の reply 連鎖が該当** ②各アカに固有の電話番号が必要（SIM4枚）＋同一端末自動化の flag リスク
- **1アカウントが議論を自分へのリプライ＝スレッドとして連投**。自己スレッドは「他者への反応」に当たらず完全に規約セーフ
- 人格の見分けは各投稿頭の話者名/絵文字プレフィックス（例 `🟢 Optimist:` / `🔴 Skeptic:` / `🟣 Zen:` / `▣ Host:`）で表現

### 実装要点（着手前メモ）
- フック点: `turn-runner.ts` `advanceOneTurn` の `addMessage` 成功直後に当該ターンを投稿
- スレッド連結: AT Protocol の reply は root と parent の {uri, cid} 参照が必須 → 各メッセージの post ref を保存（messages 列追加 or meta に議題別保存。設計で確定）
- リンク: Bluesky は素のURLを自動リンクしない → **facets（バイト範囲）で明示**。root（turn1）または最終投稿に議題ページURLを付与
- 認証: app password を `BLUESKY_IDENTIFIER` / `BLUESKY_APP_PASSWORD` secret で（ハードコード禁止）。OAuth は dev preview のため当面 app password 採用
- 安全弁: `bluesky_enabled` メタフラグで再デプロイなし停止（TTS予算ガードと同パターン）。投稿は best-effort、失敗しても議論cronは止めない
- 冪等性: cron 二重発火・リトライでも同ターンを二重投稿しない（post ref 保存済みならスキップ）
